# Câu hỏi #414

Which design principles for cloud architecture are recommended when re-architecting a large monolithic application? (Select TWO)

## Options

**A.** Use manual monitoring.
**B.** Use fixed servers.
**C.** Implement loose coupling.
**D.** Rely on individual components.
**E.** Design for scalability.


---

## 1. CONTEXT & ĐỀ BÀI
- **Scenario:** Tái kiến trúc một ứng dụng monolithic lớn sang mô hình cloud-native.
- **Existing Resources:** Một ứng dụng monolithic lớn (single-tier, tight coupling).
- **Current Issue/Goal:** Áp dụng design principle phù hợp cho cloud architecture.

## 2. KEYWORDS QUAN TRỌNG
| Keyword | Ý nghĩa / Gợi ý |
|---------|-----------------|
| `re-architecting a large monolithic application` | Phá vỡ monolith thành các service nhỏ hơn |
| `loose coupling` | Các component độc lập, một component fail không ảnh hưởng component khác |
| `design for scalability` | Hệ thống có thể scale theo nhu cầu (horizontal/vertical) |
| `manual monitoring` | Ngược lại với automation — không phải best practice |
| `fixed servers` | Trái với elasticity của cloud |
| `rely on individual components` | Single point of failure — không khuyến khích |

## 3. YÊU CẦU CỦA ĐỀ
- **Question type:** Cloud architecture design principles (multi-answer)
- **Constraints:** Phù hợp với cloud-native khi phá vỡ monolith

## 4. ĐÁP ÁN ĐÚNG
**✅ Đáp án: C và E**

**Giải thích:**
- `Loose coupling` (Implement loose coupling): khi tách monolith, các microservice nên giao tiếp qua API/queue/event thay vì gọi trực tiếp, giúp hệ thống chịu lỗi tốt hơn và triển khai độc lập.
- `Design for scalability`: cloud cho phép scale theo nhu cầu thực tế. Kiến trúc mới phải tận dụng `Auto Scaling`, `Elastic Load Balancing`, decoupling qua `SQS`/`SNS`/`EventBridge`.
- Cả hai principle cùng nhau giúp ứng dụng cloud-native đạt reliability và performance.

## 5. CÁC ĐÁP ÁN SAI
**❌ Đáp án A:**
- `Manual monitoring` trái với design principle "Implement automation" của AWS Well-Architected.
- Đáp án này nên thay bằng `automated monitoring` qua CloudWatch, CloudTrail, X-Ray.

**❌ Đáp án B:**
- `Fixed servers` trái với nguyên tắc elasticity. Trong cloud, nên dùng Auto Scaling + Spot/On-Demand mix thay vì giữ server cố định.
- Đáp án này là anti-pattern cho cloud architecture.

**❌ Đáp án D:**
- "Rely on individual components" tạo single point of failure. Nguyên tắc đúng là "design for failure" — assume mọi component có thể chết và xây dựng cơ chế recovery.
- Đáp án này đi ngược lại với cloud best practice.

## 6. MẸO GHI NHỚ (Memory Hook)
🧠 *Cloud design cho monolith-to-microservice: `loose coupling` + `scalability` + `design for failure` + `automation`. Tránh: `tight coupling`, `fixed servers`, `manual`.*
