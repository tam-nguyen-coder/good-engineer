# Question #507 - Topic 1

A company has a web application for travel ticketing. The application is based on a database that runs in a single data center in North America. The company wants to expand the application to serve a global user base. The company needs to deploy the application to multiple AWS Regions. Average latency must be less than 1 second on updates to the reservation database. The company wants to have separate deployments of its web platform across multiple Regions. However, the company must maintain a single primary reservation database that is globally consistent. Which solution should a solutions architect recommend to meet these requirements?

## Options

**A.** Convert the application to use Amazon DynamoDB. Use a global table for the center reservation table. Use the correct Regional endpoint in each Regional deployment.

**B.** Migrate the database to an Amazon Aurora MySQL database. Deploy Aurora Read Replicas in each Region. Use the correct Regional endpoint in each Regional deployment for access to the database.

**C.** Migrate the database to an Amazon RDS for MySQL database. Deploy MySQL read replicas in each Region. Use the correct Regional endpoint in each Regional deployment for access to the database.

**D.** Migrate the application to an Amazon Aurora Serverless database. Deploy instances of the database to each Region. Use the correct Regional endpoint in each Regional deployment to access the database. Use AWS Lambda functions to process event streams in each Region to synchronize the databases.

## 1. CONTEXT & ĐỀ BÀI
- **Scenario:** Travel ticketing app mở rộng global. Cần deploy multi-region, <1s latency on updates, single primary database globally consistent.
- **Existing Resources:** Single data center database in North America.
- **Current Issue/Goal:** Global deployment, low-latency writes, strong consistency.

## 2. KEYWORDS QUAN TRỌNG
| Keyword | Ý nghĩa / Gợi ý |
|---------|-----------------|
| `globally consistent` | Strong consistency, eventual consistency không đáp ứng. |
| `single primary reservation database` | Một database chính duy nhất (single writer). |
| `less than 1 second on updates` | Update nhanh, nhưng vẫn consistent. |
| `separate deployments across multiple Regions` | Multi-region web + read replicas local. |

## 3. YÊU CẦU CỦA ĐỀ
- **Question type:** Meet requirements
- **Constraints:** Globally consistent, single primary, <1s latency on updates, multi-region

## 4. ĐÁP ÁN ĐÚNG
**✅ Đáp án: B**

**Giải thích:**
- **Đảm bảo tính nhất quán toàn cầu với Single Primary (Đơn ghi):** Đề bài yêu cầu *"maintain a single primary reservation database that is globally consistent"* (duy trì một cơ sở dữ liệu đặt chỗ chính duy nhất có tính nhất quán toàn cầu để tránh tình trạng trùng lặp/overbooking). Amazon Aurora MySQL sử dụng kiến trúc Single-Master (chỉ có duy nhất một Primary Writer Instance nhận các lệnh Ghi/Ghi đè trong vùng chính, các vùng khác là Read Replicas). Điều này đảm bảo tính nhất quán mạnh mẽ (strong consistency) cho các giao dịch đặt vé.
- **Thời gian đồng bộ cực nhanh (< 1 giây):** Cơ chế nhân bản của **Amazon Aurora Global Database** hoạt động ở tầng lưu trữ vật lý (storage layer), không phụ thuộc vào cơ chế log replication của database engine. Điều này giúp độ trễ sao chép dữ liệu (replication lag) giữa các Region cực kỳ thấp (thường < 100ms), hoàn toàn đáp ứng yêu cầu *"latency must be less than 1 second on updates"*.
- **Tách biệt Đọc/Ghi giữa các vùng:** Nền tảng web ở mỗi Region có thể đọc dữ liệu đặt chỗ ngay lập tức từ Aurora Read Replicas tại Region đó để tối ưu hiệu năng đọc, trong khi các lệnh Ghi (đặt vé) vẫn được gửi về Primary Writer ở Bắc Mỹ thông qua cơ chế định tuyến hoặc Write Forwarding.

## 5. CÁC ĐÁP ÁN SAI
**❌ Đáp án A:**
- **Amazon DynamoDB Global Tables:** Mặc dù DynamoDB rất nhanh, nhưng Global Tables hoạt động theo mô hình **multi-active/multi-master** (cho phép ghi dữ liệu đồng thời ở bất kỳ Region nào). Do đó, nó không đáp ứng yêu cầu duy trì một *"single primary reservation database"*. Ngoài ra, cơ chế đồng bộ của DynamoDB Global Tables là bất đồng bộ (eventual consistency) với cơ chế giải quyết xung đột "Last-Writer-Wins", điều này rất dễ dẫn đến lỗi bán trùng vé (double booking) trong hệ thống đặt chỗ.

**❌ Đáp án C:**
- **RDS for MySQL Read Replicas:** RDS MySQL truyền thống sử dụng cơ chế sao chép dựa trên binlog (logical replication) ở tầng cơ sở dữ liệu. Cơ chế này chậm hơn đáng kể so với bản sao lưu ở tầng lưu trữ của Aurora, dễ gặp hiện tượng trễ sao chép lớn (high replication lag) vượt quá 1 giây dưới tải cao và không có tính năng tự động failover/quản trị toàn cầu tốt như Aurora.

**❌ Đáp án D:**
- **Aurora Serverless + Lambda đồng bộ:** Việc tự viết AWS Lambda để đồng bộ dữ liệu giữa các Region độc lập là giải pháp cực kỳ phức tạp, tạo ra gánh nặng vận hành lớn và chỉ đạt được tính nhất quán cuối cùng (eventual consistency), không đảm bảo tính nhất quán giao dịch toàn cầu.

## 6. MẸO GHI NHỚ (Memory Hook)
🧠 *"Hệ thống đặt vé (ticketing/booking) yêu cầu Global Consistency + Single Writer → Chọn Aurora Global Database (Single Primary + Replicas). Tránh dùng DynamoDB Global Tables vì là Multi-Master và chỉ có Eventual Consistency."*
