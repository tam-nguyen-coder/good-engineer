# Question #439 - Topic 1

A solutions architect configured a VPC that has a small range of IP addresses. The number of Amazon EC2 instances that are in the VPC is increasing, and there is an insufficient number of IP addresses for future workloads. Which solution resolves this issue with the LEAST operational overhead?

## Options

**A.** Add an additional IPv4 CIDR block to increase the number of IP addresses and create additional subnets in the VPC. Create new resources in the new subnets by using the new CIDR.

**B.** Create a second VPC with additional subnets. Use a peering connection to connect the second VPC with the first VPC Update the routes and create new resources in the subnets of the second VPC.

**C.** Use AWS Transit Gateway to add a transit gateway and connect a second VPC with the first VPUpdate the routes of the transit gateway and VPCs. Create new resources in the subnets of the second VPC.

**D.** Create a second VPC. Create a Site-to-Site VPN connection between the first VPC and the second VPC by using a VPN-hosted solution on Amazon EC2 and a virtual private gateway. Update the route between VPCs to the traffic through the VPN. Create new resources in the subnets of the second VPC.

## 1. CONTEXT & ĐỀ BÀI
- **Scenario:** VPC small IP range, running out of IPs for EC2 instances.
- **Existing Resources:** VPC with small CIDR block.
- **Current Issue/Goal:** More IP addresses. Least operational overhead.

## 2. KEYWORDS QUAN TRỌNG
| Keyword | Ý nghĩa / Gợi ý |
|---------|-----------------|
| `VPC` | Can add secondary CIDR blocks. |
| `additional IPv4 CIDR block` | Extend VPC IP range without new VPC. |
| `least operational overhead` | Add CIDR to existing VPC (no peering/transit needed). |

## 3. YÊU CẦU CỦA ĐỀ
- **Question type:** Networking
- **Constraints:** More IPs, least overhead

## 4. ĐÁP ÁN ĐÚNG
**✅ Đáp án: A**

**Giải thích:**
- **Thêm secondary CIDR block:** AWS cho phép liên kết thêm các khối CIDR IPv4 phụ (secondary CIDR blocks) vào một VPC đã tồn tại để mở rộng không gian địa chỉ IP mà không làm gián đoạn các dịch vụ đang chạy. Sau khi thêm CIDR mới, Solutions Architect chỉ cần tạo thêm các subnets mới nằm trong dải CIDR này để triển khai các EC2 instances tiếp theo.
- **Least Operational Overhead:** Đây là giải pháp đơn giản nhất vì tất cả tài nguyên vẫn nằm chung trong một VPC đơn lẻ. Không cần thiết lập thêm các kết nối mạng phức tạp, giữ nguyên kiến trúc bảo mật (Security Groups, Network ACLs) và không phải chỉnh sửa cấu hình định tuyến liên mạng.

## 5. CÁC ĐÁP ÁN SAI
**❌ Đáp án B:**
- **VPC Peering:** Việc tạo VPC thứ hai và kết nối với VPC cũ thông qua VPC Peering đòi hỏi phải cấu hình kết nối Peering, cập nhật bảng định tuyến (Route Tables) ở cả hai bên VPC để cho phép lưu lượng đi qua peering link. Việc này tạo ra nhiều gánh nặng cấu hình và quản trị hơn so với việc mở rộng dải IP trong cùng một VPC.

**❌ Đáp án C:**
- **AWS Transit Gateway:** Transit Gateway là dịch vụ trung chuyển để kết nối hàng chục/hàng trăm VPCs hoặc mạng on-premises. Sử dụng Transit Gateway chỉ để kết nối 2 VPCs nhằm giải quyết vấn đề thiếu IP là giải pháp quá phức tạp (overkill) và tốn kém chi phí không đáng có.

**❌ Đáp án D:**
- **Site-to-Site VPN tự host trên EC2:** Việc tự triển khai và quản lý giải pháp VPN (Software VPN) trên EC2 làm tăng rất lớn gánh nặng vận hành (cấu hình VPN software, vá lỗi bảo mật, quản lý tính sẵn sàng của EC2 VPN...) đồng thời phải cấu hình Virtual Private Gateway (VGW) và định tuyến lại toàn bộ lưu lượng qua VPN. Đây là giải pháp phức tạp nhất và có hiệu năng kém nhất.

## 6. MẸO GHI NHỚ (Memory Hook)
🧠 *"Need more IPs in VPC → add secondary CIDR block. Peering/Transit = more overhead."*
