# Question #447 - Topic 1

A company has a stateless web application that runs on AWS Lambda functions that are invoked by Amazon API Gateway. The company wants to deploy the application across multiple AWS Regions to provide Regional failover capabilities. What should a solutions architect do to route traffic to multiple Regions?

## Options

**A.** Create Amazon Route 53 health checks for each Region. Use an active-active failover configuration.

**B.** Create an Amazon CloudFront distribution with an origin for each Region. Use CloudFront health checks to route traffic.

**C.** Create a transit gateway. Attach the transit gateway to the API Gateway endpoint in each Region. Configure the transit gateway to route requests.

**D.** Create an Application Load Balancer in the primary Region. Set the target group to point to the API Gateway endpoint hostnames in each Region.

## 1. CONTEXT & ĐỀ BÀI
- **Scenario:** Stateless web app: API Gateway + Lambda. Deploy across Regions for failover.
- **Existing Resources:** API Gateway, Lambda.
- **Current Issue/Goal:** Route traffic across multiple Regions.

## 2. KEYWORDS QUAN TRỌNG
| Keyword | Ý nghĩa / Gợi ý |
|---------|-----------------|
| `multiple AWS Regions` | Cross-region traffic routing. |
| `Regional failover` | Route 53 active-active with health checks. |
| `stateless` | Any Region can handle any request. |

## 3. YÊU CẦU CỦA ĐỀ
- **Question type:** Global / DNS routing
- **Constraints:** Multi-Region failover for API Gateway + Lambda

## 4. ĐÁP ÁN ĐÚNG
**✅ Đáp án: A**

**Giải thích:**
- **Route 53 Active-Active Failover:** Trong cấu hình active-active, Amazon Route 53 định tuyến lưu lượng truy cập của người dùng đến tất cả các Region đang hoạt động tốt (healthy) cùng một lúc (thường sử dụng chính sách định tuyến Latency hoặc Weighted).
- **Health Checks:** Bằng cách thiết lập các Route 53 health checks trỏ trực tiếp đến các endpoint API Gateway ở từng Region, Route 53 sẽ liên tục giám sát sức khỏe của từng khu vực. Nếu một Region gặp sự cố (unhealthy), Route 53 sẽ tự động loại bỏ bản ghi DNS của Region đó khỏi kết quả phản hồi DNS. Lưu lượng người dùng sẽ ngay lập tức được điều hướng sang các Region lành lặn còn lại, đảm bảo tính sẵn sàng cao và khả năng tự động khôi phục (Regional failover).

## 5. CÁC ĐÁP ÁN SAI
**❌ Đáp án B:**
- **CloudFront Origin Failover:** CloudFront có tính năng Origin Group để failover giữa các origins, nhưng đây là cơ chế active-passive (chỉ failover khi origin chính trả về mã lỗi HTTP 5xx) chứ không phải active-active. Ngoài ra, CloudFront không có tính năng gọi là "CloudFront health checks" dùng để tự định tuyến lưu lượng giữa các Region như Route 53.

**❌ Đáp án C:**
- **AWS Transit Gateway:** Là dịch vụ kết nối mạng ở tầng Layer 3 (Network) dùng để liên kết các VPCs và mạng on-premises trong kiến trúc Hub-and-Spoke. Transit Gateway không thể tiếp nhận và định tuyến các truy vấn HTTP/HTTPS công cộng từ internet đến các endpoint của API Gateway.

**❌ Đáp án D:**
- **ALB ở Primary Region:** ALB là dịch vụ hoạt động ở cấp Region (Regional service). Nếu đặt ALB ở Region chính, khi Region chính gặp sự cố (failover xảy ra), ALB này cũng sẽ ngừng hoạt động, khiến hệ thống mất hoàn toàn khả năng định tuyến. Hơn nữa, ALB target group không thể cấu hình trực tiếp các hostname công cộng của API Gateway ở Region khác làm target để định tuyến cross-region.

## 6. MẸO GHI NHỚ (Memory Hook)
🧠 *"Multi-Region API failover → Route 53 active-active + health checks. ALB và Transit Gateway là dịch vụ Regional/Internal, không dùng cho Global DNS routing."*
