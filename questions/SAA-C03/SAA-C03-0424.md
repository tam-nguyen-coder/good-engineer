# Question #424 - Topic 1

A company is running a custom application on Amazon EC2 On-Demand Instances. The application has frontend nodes that need to run 24 hours a day, 7 days a week and backend nodes that need to run only for a short time based on workload. The number of backend nodes varies during the day. The company needs to scale out and scale in more instances based on workload. Which solution will meet these requirements MOST cost-effectively?

## Options

**A.** Use Reserved Instances for the frontend nodes. Use AWS Fargate for the backend nodes.

**B.** Use Reserved Instances for the frontend nodes. Use Spot Instances for the backend nodes.

**C.** Use Spot Instances for the frontend nodes. Use Reserved Instances for the backend nodes.

**D.** Use Spot Instances for the frontend nodes. Use AWS Fargate for the backend nodes.

## 1. CONTEXT & ĐỀ BÀI
- **Scenario:** Frontend: 24/7. Backend: short-lived, variable load. Most cost-effective.
- **Existing Resources:** EC2 On-Demand instances.
- **Current Issue/Goal:** Optimize cost: RI for steady, Spot for variable.

## 2. KEYWORDS QUAN TRỌNG
| Keyword | Ý nghĩa / Gợi ý |
|---------|-----------------|
| `24 hours a day, 7 days a week` | Reserved Instances: cost saving cho workload ổn định. |
| `short time based on workload` | Spot Instances: rẻ nhất (60-90%), có thể bị interrupt. |
| `most cost-effectively` | RI (steady) + Spot (variable). |

## 3. YÊU CẦU CỦA ĐỀ
- **Question type:** Cost optimization
- **Constraints:** Frontend 24/7, backend short/variable

## 4. ĐÁP ÁN ĐÚNG
**✅ Đáp án: B**

**Giải thích:**
- **Frontend chạy 24/7 (Workload ổn định):** Sử dụng **Reserved Instances (RIs)** là tối ưu nhất vì frontend chạy liên tục không ngừng nghỉ. Việc cam kết sử dụng 1 năm hoặc 3 năm với RIs sẽ giúp giảm chi phí lên đến 72% so với giá On-Demand.
- **Backend chạy ngắn hạn và thay đổi theo khối lượng công việc (Workload linh hoạt):** Sử dụng **Spot Instances** cho phép tận dụng lượng tài nguyên EC2 dư thừa của AWS với mức giá cực rẻ (giảm tới 90% so với On-Demand). Vì các node backend được thiết kế để tự động scale out/scale in tùy theo tải, chúng là đối tượng hoàn hảo cho mô hình Spot (vốn có thể bị AWS thu hồi bất kỳ lúc nào với cảnh báo trước 2 phút).
- **Yêu cầu hệ thống:** Đề bài nhấn mạnh việc chạy ứng dụng trên **Amazon EC2** và cần scale thêm/giảm bớt **instances** (máy chủ ảo), do đó việc tiếp tục tối ưu hóa trên nền tảng EC2 bằng sự kết hợp giữa RI + Spot là giải pháp trực tiếp và tối ưu nhất về chi phí.

## 5. CÁC ĐÁP ÁN SAI
**❌ Đáp án A:**
- **AWS Fargate cho backend:** Fargate là dịch vụ serverless compute dành cho containers (ECS/EKS). Đề bài đang chạy ứng dụng tùy chỉnh trực tiếp trên các EC2 instances và không đề cập đến việc container hóa ứng dụng. Ngoài ra, chi phí chạy Fargate thông thường vẫn cao hơn đáng kể so với việc dùng Spot Instances trên EC2.

**❌ Đáp án C:**
- **Spot cho frontend và RI cho backend:** Đưa Spot Instances vào chạy frontend 24/7 là cực kỳ rủi ro vì nếu AWS thu hồi instance, frontend sẽ bị gián đoạn. Ngược lại, dùng RIs cho backend (vốn chỉ chạy ngắn hạn và có số lượng node biến động liên tục) sẽ gây lãng phí rất lớn vì bạn phải trả tiền cho các Reserved Instances ngay cả khi chúng không hoạt động.

**❌ Đáp án D:**
- **Spot cho frontend:** Tương tự C, việc dùng Spot cho frontend là sai nguyên tắc thiết kế hệ thống có tính sẵn sàng cao (High Availability) cho các dịch vụ chạy liên tục. Fargate cho backend cũng không tối ưu chi phí bằng Spot Instances.

## 6. MẸO GHI NHỚ (Memory Hook)
🧠 *"Steady 24/7 → Reserved. Variable/short → Spot. Spot for fault-tolerant workloads."*
