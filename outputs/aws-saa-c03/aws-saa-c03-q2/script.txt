AWS Certified Solutions Architect Associate, question two. A company needs the ability to analyze the log files of its proprietary application. The logs are stored in JSON format in an Amazon S3 bucket. Queries will be simple and will run on-demand. A solutions architect needs to perform the analysis with minimal changes to the existing architecture. What should the solutions architect do to meet these requirements with the LEAST amount of operational overhead?

Let's review the options.
Option A: Use Amazon Redshift to load all the content into one place and run the SQL queries as needed.
Option B: Use Amazon CloudWatch Logs to store the logs. Run SQL queries as needed from the Amazon CloudWatch console.
Option C: Use Amazon Athena directly with Amazon S3 to run the queries as needed.
Option D: Use AWS Glue to catalog the logs. Use a transient Apache Spark cluster on Amazon EMR to run the SQL queries as needed.

The correct answer is Option C.
Option C is correct because Amazon Athena is a serverless query service that allows you to analyze data directly in Amazon S3 using standard SQL. There is no need to load or transform the data, or spin up any infrastructure, minimizing changes and operational overhead.

Option A is incorrect. Amazon Redshift requires provisioned data warehouses or serverless endpoints, and you must load the data into tables. This introduces significant operational overhead and ETL design.
Option B is incorrect. Moving logs from Amazon S3 to CloudWatch Logs would require changing the application's logging target or running an import process, violating the minimal changes requirement.
Option D is incorrect. Running a transient EMR Spark cluster requires configuring cluster settings, managing EMR instances, and waiting for startup, which adds heavy operational complexity.

Here is your memory hook. S3 data lake plus on-demand SQL queries equals Amazon Athena. Avoid Redshift ETL, CloudWatch migrations, or EMR clusters.
