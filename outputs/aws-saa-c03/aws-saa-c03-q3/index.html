The company wants to limit access to this S3 bucket to only users of accounts within the organization. Which solution meets these requirements with the LEAST amount of operational overhead?

ANSWER DEEP DIVE Detailed justifications
Technical Deep Dive
A
CORRECT ANSWER

Add aws:PrincipalOrgID condition key to S3 bucket policy...

PrincipalOrgID: Restricts access to all accounts inside the organization dynamically.
Zero Maintenance: Automatically handles new or deleted accounts in AWS Organizations.
Least Overhead: Single S3 bucket policy change, no tagging or code required.
B
INCORRECT

Create OUs and use aws:PrincipalOrgPaths...

OU Paths: Restricting by OU path adds unnecessary department boundary mapping.
Higher Overhead: Adds path structural requirements when whole org access was requested.
C
INCORRECT

Monitor CloudTrail events and update policy dynamically...

Over-engineered: Monitoring API events and executing automated updates is complex.
Fragile: Event delays or failures could leave bucket policies insecure or broken.
D
INCORRECT

Tag users and use aws:PrincipalTag...

Not Scalable: Tagging all users across member accounts requires constant manual effort.
Cross-Account Complexity: Session tags must be explicitly trusted and forwarded.
